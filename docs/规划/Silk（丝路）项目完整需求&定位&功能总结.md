# Silk（丝路）项目完整需求&定位&功能总结

## 一、项目名称

**Silk｜丝路**

寓意：数字丝路，打通各大AI模型壁垒，让AI数据流自由流转、互通、中转、贯通。

## 二、产品定位（核心定稿）

**Silk 是一款纯本地桌面客户端 AI 多模型统一中转站**，无网页、无服务端、不上云，轻量化常驻后台。

目标：**本地仅需一个统一接口，即可无缝调用全网所有大模型、绘图、视频AI接口**，彻底解决多模型协议不统一、客户端不兼容、切换繁琐的问题。

区别于所有 Web 网关、服务器中转工具，Silk 专注 **个人本地桌面私有化使用**。

## 三、核心需求（为什么要做）

- 三套核心LLM协议范式（OpenAIChat、Claude Messages、OpenAI Response）格式不统一、出入站规范割裂，各类AI客户端适配混乱、无法通用互通

- 图像、视频生成类AI接口分散、无统一入口，无法统一管理

- 各类AI客户端（Cursor、Chatbox、绘图工具）只能绑定单一模型，切换必须改配置、换密钥

- 现有开源工具要么是网页后台、要么是功能残缺、要么是Electron笨重臃肿

- 用户需要**纯本地、可视化、可常驻、零云端上传、高性能**的私有中转工具

## 四、核心能力（三大王牌功能，项目核心壁垒）

### 1. 三套核心协议双向归一互转（核心独有功能）

基于项目定制三套核心协议体系，实现双向互通、统一出站：OpenAIChat请求协议、Claude请求协议、OpenAI Response统一响应协议

- 兼容 SSE 流式输出、多模态图片输入、FunctionCall 工具调用

- 自动适配 temperature、max_tokens、上下文、参数映射

- 任意客户端以任意协议接入，可自动转为目标模型协议

### 2. 图像/视频接口 原生透传转发

对 SD、MJ、文生视频、以及所有自定义媒体接口，**不解析、不转换、原样透传**

- 完整保留 Header、Body、二进制流、文件上传

- 跳过LLM协议转换逻辑，纯代理高速转发

### 3. 本地统一单一入口

所有AI工具只连接本地统一地址：`http://127.0.0.1:xxxx`

一键切换所有模型，无需修改任何客户端配置、无需暴露上游密钥。

## 五、完整功能列表

### ✅ 桌面客户端基础能力

- 独立Tauri桌面窗口，纯客户端、无浏览器依赖

- 系统托盘后台驻留、最小化后台运行、开机自启

- 网关一键启停、端口配置、深色/浅色主题

### ✅ 模型服务商管理

- 可视化添加/编辑/删除 LLM/图像/视频 模型渠道

- 支持独立代理、超时、密钥、模型别名、状态开关

- 实时连通性测试、渠道可用性检测

### ✅ 协议转换引擎

- 实现三套核心协议双向归一转换：任意模型入站协议（OpenAI Chat / Claude Messages）互相转换，**所有模型统一输出 OpenAI Response 标准格式**

- 自定义模型名映射、参数兼容规则

- 单独开关控制协议转换/透传模式

### ✅ 媒体透传引擎

- 自定义路径匹配规则

- 图片、视频、二进制文件完整透传转发

### ✅ 日志与调试系统

- 全量请求/响应日志本地持久化

- JSON格式化、折叠、复制、报错高亮

- 耗时、状态码、模型来源、检索分页

### ✅ 安全与配置

- 所有密钥、规则、日志**纯本地存储**，零云端上传

- 本地网关独立鉴权密钥，隔离上游原始Key

- 配置一键导入/导出、备份恢复

## 六、技术栈最终定稿

**后端核心：Rust + Tauri + Axum + Tokio**

**数据持久：SQLx + SQLite**

**前端界面：Vue3 + NaiveUI + Tailwind**

**构建工具：Bun + Vite**

**核心特性：轻量、高性能、低内存、跨平台、无内存泄漏、长期挂机稳定**

## 七、产品差异化（核心优势）

- 对比 OneAPI/LiteLLM：**纯桌面客户端，无需网页、无需部署**

- 对比 Amux：**Tauri 极致轻量，无Electron臃肿，完全自研可控内核**

- 对比各类小工具：**唯一同时具备三模型双向互转 + 全媒体透传的本地GUI工具**
