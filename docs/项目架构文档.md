## 1. 项目概述

Silk（丝路）是一个纯本地的桌面 AI 多模型中转网关客户端。它：

- 在本地启动 HTTP 网关服务（默认 `127.0.0.1:9876`）
- 接收符合 OpenAI Chat / Claude Messages / OpenAI Response 协议的请求
- 通过协议适配器实现三大协议的任意互转（手写适配器，不使用第三方库）
- 将请求路由到上游 AI 提供方（OpenAI、Claude、通义千问等）
- 提供 Vue 3 桌面 UI 管理 Provider、路由规则、日志、模型映射等

### 核心功能

| 功能 | 描述 |
|------|------|
| 协议双向转换 | 三大 LLM 协议（OpenAI Chat / Claude Messages / OpenAI Response）任意互转 |
| 本地网关 | 本机 HTTP 服务，外部 AI 工具直连 `127.0.0.1` 即可使用 |
| 流式响应 | 完整支持 SSE 流式输出，心跳保活（15s），超时断连（30s），Last-Event-ID 续传 |
| 图片/视频 AI 透传 | 非 LLM 请求直接转发到上游提供方 |
| Provider 管理 | 支持多家 AI 服务提供商统一管理，多 Key 负载均衡 |
| 路由规则 | 按 Host/Path/Method/ContentType 匹配路由，直接转发到目标 Provider，兼容历史 `target_group_id` |
| 模型池 | 模型映射管理，支持渠道分配、负载均衡策略、计费信息 |
| 请求日志 | 完整的请求/响应日志记录、查询、统计、CSV 导出 |
| 三级失败回退 | Level 1 重试 → Level 2 换 Key → Level 3 换渠道 |
| 本地存储 | 所有数据存于 SQLite，零云端依赖 |
| 网关认证 | 基于 `sk-gw-` 前缀的 API Key 认证，SHA-256 哈希验证 |
| 限流控制 | 按客户端 IP 的请求数/Token 数限流 |

## 2. 技术栈

| 层级 | 技术 | 用途 |
|------|------|------|
| **UI 框架** | Vue 3 (Composition API) + TypeScript | 桌面界面 |

## 7. 网关中间件管道

### 7.1 管道流程（10 阶段主流程）

所有外部 HTTP 请求 `→` 经过 Axum Fallback 路由 `→` GatewayPipeline `→` 10 个主阶段处理，日志写入在 `finalize_with_log()` 中异步完成：

```
                        ┌──────────────────┐
                        │   HTTP Request   │
                        └────────┬─────────┘
                                 │
                        ┌────────▼─────────┐
                        │  1. extract      │   提取请求头/路径/查询参数
                        │  initialize()    │   生成 request_id (UUID)
                        │  read_body()     │   读取请求体（2MB 限制）
                        └────────┬─────────┘
                                 │
                        ┌────────▼─────────┐
                        │  2. authenticate │   验证 sk-gw-* 密钥
                        │  run()           │   SHA-256 哈希匹配
                        │                  │   仅在 /v1/* 路径生效
                        └────────┬─────────┘
                                 │
                        ┌────────▼─────────┐
                        │  3. rate_limit   │   按客户端 IP 限流
                        │  run()           │   请求数/min + Token/min
                        │                  │   支持 per-provider 隔离
                        └────────┬─────────┘
                                 │
                        ┌────────▼─────────┐
                        │  4. resolve_     │   匹配路由规则
                        │  route::run()    │   → 解析目标 Provider
                        │                  │   → 或直接返回 /v1/models
                        └────────┬─────────┘
                                 │
               ┌─────────────────┼──────────────────┐
               │                                  │
               │            ┌──────────────────────▼─────────┐
               │            │  5. select_channel::run()      │
               │            │  按 Key 策略选择上游 Key       │
               │            │  跳过已失败的 Key              │
               │            └──────────────────────┬─────────┘
               │                                  │
               │            ┌──────────────────────▼─────────┐
               │            │  6. transform_request::run()   │
               │            │  入站协议 → 上游协议转换       │
               │            │  由 ProviderAdapter 完成        │
               │            └──────────────────────┬─────────┘
               │                                  │
               │            ┌──────────────────────▼─────────┐
               │            │  7. dispatch_upstream::run()   │
               │            │  转发到上游 Provider           │
               │            │  SSE 流式/非流式 自动判断       │
               │            │  指数退避重试（最多 3 次）     │
               │            │  返回完整响应或 SSE 流          │
               │            └──────────────────────┬─────────┘
               │                                  │
               │                                  │ ← 失败 → Level 2 换 Key / Level 3 换渠道
               │            ┌──────────────────────▼─────────┐
               │            │  8. transform_response::run()  │
               │            │  上游协议 → 出站协议转换       │
               │            └──────────────────────┬─────────┘
               │                                  │
               │            ┌──────────────────────▼─────────┐
                │            │  9. finalize_with_log()         │
               │            │  异步发送日志到 channel         │
               │            └──────────────────────┬─────────┘
               │                                  │
               └──────────────────────────────────┤
                                                  │
                                     ┌────────────▼────────────┐
                                     │  finalize::success()    │
                                     │  或 finalize::failure() │
                                     │  构建最终 HTTP Response │
                                     └─────────────────────────┘
```

### 7.2 中间件详细说明

| # | 中间件 | 文件 | 职责 | 副作用 |
|---|--------|------|------|--------|
| 1 | `extract` | `extract.rs` | 解析请求头、URI、方法；读取请求体（上限 2MB）；生成 `request_id` | 写入 `RequestContext` |
| 2 | `authenticate` | `authenticate.rs` | 检查 `/v1/*` 路径的 `Authorization: Bearer <sk-gw-xxx>`；SHA-256 哈希匹配 `gateway_keys` 表 | 注入 `auth_key_name` |
| 3 | `rate_limit` | `rate_limit.rs` | 按客户端 IP + Provider ID 进行请求数/Token 数限流 | 拒绝时返回 429 |
| 4 | `resolve_route` | `resolve_route.rs` | 按 Host/Path/Method/ContentType 匹配 `routing_rules`；解析目标 Provider；支持 `/v1/models` 返回 Provider 模型列表 | 设置 `route`, `provider`, `channels_available` |
| - | **`[before_route]`** | `plugin.rs` | **插件钩子**：路由后立刻执行。可用于缓存直接命中拦截响应，跳过后续所有步骤 | 可注入 `response` 截断请求 |
| 5 | `select_channel` | `select_channel.rs` | 按 Provider 的 `key_strategy` 从可用 Key 中选择；跳过 `failed_keys` | 设置 `selected_api_key` |
| 6 | `transform_request` | `transform_request.rs` | 调用适配器的 `transform_request()` 实现协议转换；构建上游 URL 和请求体 | 生成 `UpstreamRequest` |
| - | **`[before_upstream]`** | `plugin.rs` | **插件钩子**：转发上游前瞬间执行。用于 Prompt 缓存注入、日志折叠、历史窗口裁剪 | 修改 `request_body` 和上游参数 |
| 7 | `dispatch_upstream` | `dispatch_upstream.rs` | 用 Reqwest 转发请求到上游；支持流式/非流式；指数退避重试（最多 3 次）；SSE 心跳 15s/超时 30s | 调用 `stream_response` |
| 8 | `transform_response` | `transform_response.rs` | 调用适配器的 `transform_response()` 将上游响应转为客户端协议格式 | 更新 `upstream_body` |
| - | **`[after_upstream]`** | `plugin.rs` | **插件钩子**：响应格式转换后执行。用于将成功响应回填至本地缓存库 | 无 |
| 9 | `finalize_with_log` | `pipeline.rs` | 流式场景异步发送日志，非流式场景在返回前写入日志 | 写入 `RequestLog` |
| 10 | `finalize` | `finalize.rs` | 构建最终 HTTP Response；设置状态码/响应头/响应体 | 返回 `Response` |

### 7.3 插件系统与 Token 节省拦截层

网关引入了基于生命周期钩子拦截的插件系统（`GatewayPlugin`），允许在请求/响应流动的特定时刻进行操作，用以对用户无感地接入最新的 Token 节省策略。

#### 7.3.1 插件生命周期钩子
*   **`before_route`**：读取请求体后立刻执行。可向 `ctx.response` 注入响应直接截断网络访问，适用于本地缓存（Semantic Cache / Exact Cache）命中拦截。
*   **`before_upstream`**：协议转换完毕、转发上游服务器前一刻执行。可修改 `ctx.request_body` 的最终上游格式，适用于参数改写与内容折叠。
*   **`after_upstream`**：收到响应并转换为 OpenAI 兼容格式后执行。适用于成功结果的回填与审计。

#### 7.3.2 默认挂载的 Token 节省插件
目前项目默认开启了以下 3 个原生 Rust 插件：
1.  **`PromptCachePlugin` (Prompt 缓存优化器)**：
    当请求体大于 6000 字节（约 1500~2000 tokens）且目标为 Claude 时，自动在 system 提示词及倒数第 2 条历史消息的末尾注入 `"cache_control": {"type": "ephemeral"}` 缓存指令，触发上游 Prompt Cache 提高长上下文处理效率。
2.  **`SlidingWindowPlugin` (历史滑动窗口裁剪)**：
    配置最大保留轮数（默认 12 轮，即 24 条消息）。当对话长度超标时，保留首条 System 提示词，并裁剪中间的旧消息，自动插入系统截断声明，避免上下文无限膨胀。
3.  **`TerminalLogPrunerPlugin` (终端日志折叠压缩)**：
    检测用户输入中是否夹带了大段的控制台/终端构建日志（通过堆栈调用、`error:`/`warning:` 或特定构建提示进行评分）。一旦识别，将文本截断为首部 25 行（报错入口）和尾部 25 行（报错原因），中间内容折叠替换，为 IDE 编码工具提供极高比例的 token 级降噪与省钱优化。

### 7.4 三级失败回退机制

```
Level 1: dispatch_upstream 内部重试（最多 3 次，指数退避 500ms→8s）
    ↓ 失败
Level 2: select_channel 换 Key（跳过已失败的 Key）
    ↓ 所有 Key 失败
Level 3: resolve_route 换渠道（从 channels_available 选下一个 Provider）
    ↓ 所有渠道 + 所有 Key 失败
最终: 返回错误响应 "所有渠道和 Key 均已失败"
```

### 7.5 SSE 流式处理

```
上游 SSE 流
    │
    ▼
┌─────────────────┐
│ SseParser       │  解析 SSE 事件，追踪 last_event_id
│ (后台读取任务)  │  检测 [DONE] 结束标记
└────────┬────────┘  心跳检测超时（15s 无数据触发）
         │
         ▼
┌─────────────────┐
│ mpsc::channel   │  256 容量缓冲
│ (逐 chunk 推送) │  支持背压
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ StreamBody      │  构建 axum Response
│ (主线程接收)    │  Body::from_stream()
└────────┬────────┘
         │
         ▼
    HTTP Response
```

## 8. 协议适配器系统

### 8.1 ProviderAdapter Trait

```rust
#[async_trait]
pub trait ProviderAdapter: Send + Sync {
    fn provider_type(&self) -> &'static str;

    /// 将原始请求体（JSON）转为上游请求
    async fn transform_request(
        &self,
        req_body: &[u8],
        provider: &Provider,
        selected_api_key: &str,
    ) -> Result<UpstreamRequest, ProtocolError>;

    /// 将上游响应转为客户端期望的格式（JSON）
    async fn transform_response(
        &self,
        resp: &UpstreamResponse,
    ) -> Result<serde_json::Value, ProtocolError>;
}
```

### 8.2 已实现的适配器

| 适配器 | `provider_type` 标识 | `transform_request` 行为 | `transform_response` 行为 |
|--------|----------------------|-------------------------|--------------------------|
| **OpenAI Chat Adapter** | `"openai_chat"` | 原始 OpenAI Chat 格式 → 上游（不变） | 上游响应 → OpenAI Chat 格式 |
| **Claude Messages Adapter** | `"claude_messages"` | 转为 Claude Messages 格式 | Claude 响应 → 客户端期望格式 |
| **OpenAI Response Adapter** | `"openai_response"` | 原始 OpenAI Response 格式 → 上游（不变） | 上游响应 → OpenAI Response 格式 |

### 8.3 AdapterRegistry

```rust

}
```

### 8.4 协议转换流程

```
入站请求 → 根据 route.inbound_protocol 选择 Inbound Adapter
  → UpstreamRequest（URL + Headers + Body）
  → 发送到上游 Provider
  → UpstreamResponse（Status + Headers + Body）
  → 根据 route.outbound_protocol 选择 Outbound Adapter
  → 转换后的响应返回客户端
```

## 12. 认证与限流

### 12.1 网关认证

- **范围**: 仅 `/v1/*` 路径需要认证
- **方式**: `Authorization: Bearer <sk-gw-xxxxx>`
- **验证**: SHA-256 哈希匹配 `gateway_keys` 表（base64 编码）
- **Key 生命周期**: 支持过期时间、最大并发、启用/禁用
- **注意**: `crypto` 模块仅用于 SHA-256 哈希，不涉及对称加密

### 12.2 限流控制

- **粒度**: 按客户端 IP（取 `X-Forwarded-For` 首个 IP）
- **指标**: 每分钟请求数限制 + 每分钟 Token 数限制
- **隔离**: per-provider 独立限流（key = `ip:provider_id`）
- **窗口**: 滑动分钟窗口（60s 重置）
- **存储**: DashMap（全局共享，自动清理过期计数器）

## 13. 完整请求流程

### 13.1 外部请求处理

```
客户端 (curl/Postman/AI Tool)
    │ POST http://127.0.0.1:9876/v1/chat/completions
    │ Authorization: Bearer sk-gw-xxxxx
    │ Content-Type: application/json
    │ {"model":"gpt-4","messages":[{"role":"user","content":"Hello"}]}
    ▼
Axum Router (build_router)
    ├── /health → 200 OK {"status":"ok"}
    └── /* → proxy_handler → GatewayPipeline::execute()
        │
        ├── extract::initialize() + read_body()
        │   → RequestContext { request_id: "uuid", method, uri, headers, body }
        │
        ├── authenticate::run()
        │   → hash_api_key(token) → GatewayKeyRepo::find_by_hash()
        │   → ctx.auth_key_name = Some("key_name")
        │
        ├── rate_limit::run()
        │   → RateLimitState::check(client_ip, tokens)
        │
        ├── resolve_route::run()
        │   → RouteManager::resolve(path, method, content_type)
        │   → 匹配 RoutingRule → Provider → 设置 ctx.route, ctx.provider
        │   → 如果是 /v1/models → 直接返回模型列表响应
        │
        ├── run_with_failover() 三重循环:
        │   // 外层：换渠道（Level 3）
        │   loop {
        │       // 清理上次渠道的 Key 失败记录
        │       // 每换一个渠道做一次 per-provider 限流检查
        │
        │       // 中层：换 Key（Level 2）
        │       loop {
        │           select_channel::run() → 按策略选 Key
        │           transform_request::run() → 协议转换
        │           dispatch_upstream::run() → 包含 Level 1 重试
        │           │
        │           ├── 成功 → transform_response::run()
        │           │         → 返回 Ok(ctx)
        │           │
        │           └── 失败 → 记录 failed_keys
        │                     → 有可用 Key 继续中层循环
        │                     → 无可用 Key 跳出中层
        │       }
        │
        │       // 尝试换渠道
        │       → try_next_channel()
        │       → 有新渠道继续外层循环
        │       → 无新渠道返回 Err("所有渠道和 Key 均已失败")
        │   }
        │
        └── finalize::success(ctx) 或 finalize::failure(err)
            → 构建 axum Response (status, headers, body)
            → 返回 HTTP Response
```

### 14.2 类型级映射

| TypeScript 接口 | Rust 结构体 | 序列化注意事项 |
|----------------|------------|-------------|
| `Provider` | `Provider` | protocols/models/keys 为 JSON 字符串 ↔ Vec |
| `RoutingRule` | `RoutingRule` | Option 类型匹配 |
| `RequestLog` | `LogResponse` | timestamp: String, stream_enabled: i64 ↔ bool |
| `GatewaySettings` | `GatewaySettings` | allow_remote/rate_limit_enabled: i64 ↔ bool |
| `GatewayStatus` | `GatewayStatusResponse` | 包含 GatewaySettings |
| `DashboardStats` | `DashboardStatsResponse` | 含 computed 字段（success_rate, growth_rate） |
| `ModelMapping` | `ModelMappingResponse` | capabilities: JSON ↔ Vec\<String\> |
| `MappingChannelInfo` | `MappingChannelInfo` | selected_models: JSON ↔ Vec\<String\> |
| `GatewayKey` | `GatewayKeyResponse` | enabled: i64 ↔ bool, is_expired computed |

### 14.3 关键注意事项

1. **布尔字段序列化**: Rust 端使用 `i64`（0/1）存储布尔值，前端通过 `enabled != 0` 或 `enabled: bool` 处理转换
2. **JSON 数组字段**: `Provider.protocols`, `Provider.models`, `Provider.keys`, `ModelMapping.capabilities`, `MappingChannel.selected_models` 等字段在 Rust 端以 JSON 字符串存储，通过 `serde_json` 解析
3. **日期时间**: 传递到前端时转为 `String`（`.to_string()`），前端使用 `new Date()` 解析
4. **明文 Key**: `Provider.keys` 中的 `value` 字段明文存储（旧加密数据被标记为无效，需用户重新填写）
5. **错误处理**: 所有 Tauri 命令返回 `Result<T, String>`，前端通过 `try/catch` 捕获 `e.message`
6. **Provider 名称缓存**: 日志查询时通过 `provider_name_cache: HashMap<String, String>` 解析 provider_id → name，避免联表查询
