## 1. 项目概述

Silk（丝路）是一个纯本地的桌面 AI 多模型中转网关客户端。它：

- 在本地启动 HTTP 网关服务（`127.0.0.1:2013`）
- 接收符合 OpenAI Chat / Claude Messages / OpenAI Response 协议的请求
- 通过协议适配器实现协议双向转换（利用 `linguafranca` 库）
- 将请求路由到上游 AI 提供方（OpenAI、Claude、通义千问等）
- 提供 Vue 3 桌面 UI 管理 Provider、路由规则、日志、模型映射等

### 核心功能

| 功能 | 描述 |
|------|------|
| 协议双向转换 | 三大 LLM 协议（OpenAI Chat / Claude Messages / OpenAI Response）任意互转 |
| 本地网关 | 本机 HTTP 服务，外部 AI 工具直连 `127.0.0.1` 即可使用 |
| 流式响应 | 完整支持 SSE 流式输出，心跳保活，断线重连 |
| 图片/视频 AI 透传 | 非 LLM 请求直接转发到上游提供方 |
| Provider 管理 | 支持多家 AI 服务提供商统一管理，多 Key 负载均衡 |
| 路由分组 | 按 Host/Path/Method/ContentType 匹配路由，支持 Provider 分组和故障转移 |
| 模型池 | 模型映射管理，支持渠道分配、负载均衡策略、计费信息 |
| 请求日志 | 完整的请求/响应日志记录、查询、统计、导出 |
| 本地存储 | 所有数据存于 SQLite，零云端依赖 |
| 网关认证 | 基于 `sk-gw-` 前缀的 API Key 认证，SHA-256 哈希验证 |
| 限流控制 | 按客户端 IP 的请求数/Token 数限流 |

## 7. 网关中间件管道

### 7.1 管道流程（7 阶段）

所有外部 HTTP 请求 `→` 经过 Axum Fallback 路由 `→` GatewayPipeline `→` 7 个阶段处理：

```
                        ┌──────────────────┐
                        │   HTTP Request   │
                        └────────┬─────────┘
                                 │
                        ┌────────▼─────────┐
                        │  1. extract      │   提取请求头/路径/查询参数
                        │  initialize()    │   生成 request_id (UUID)
                        │  read_body()     │   读取请求体（2MB 限制）
                        └────────┬─────────┘
                                 │
                        ┌────────▼─────────┐
                        │  2. authenticate │   验证 sk-gw-* 密钥
                        │  run()           │   SHA-256 哈希匹配
                        │                  │   仅在 /v1/* 路径生效
                        └────────┬─────────┘
                                 │
                        ┌────────▼─────────┐
                        │  3. rate_limit   │   按客户端 IP 限流
                        │  run()           │   请求数/min + Token/min
                        │                  │   支持 per-provider 隔离
                        └────────┬─────────┘
                                 │
                        ┌────────▼─────────┐
                        │  4. resolve_     │   匹配路由规则
                        │  route::run()    │   → 解析目标 Provider
                        │                  │   → 或直接返回 /v1/models
                        │                  │     自带三级失败回退
                        └────────┬─────────┘
                                 │
               ┌─────────────────┼──────────────────┐
               │                                  │
               │            ┌──────────────────────▼─────────┐
               │            │  5. select_channel::run()      │
               │            │  按 Key 策略选择上游 Key       │
               │            │  跳过已失败的 Key              │
               │            └──────────────────────┬─────────┘
               │                                  │
               │            ┌──────────────────────▼─────────┐
               │            │  6. transform_request::run()   │
               │            │  入站协议 → 上游协议转换       │
               │            │  由 ProviderAdapter 完成        │
               │            └──────────────────────┬─────────┘
               │                                  │
               │            ┌──────────────────────▼─────────┐
               │            │  7. dispatch_upstream::run()   │
               │            │  转发到上游 Provider           │
               │            │  SSE 流式/非流式 自动判断       │
               │            │  指数退避重试（最多 3 次）     │
               │            │  返回完整响应或 SSE 流          │
               │            └──────────────────────┬─────────┘
               │                                  │
               │                                  │ ← 失败 → 换 Key / 换渠道重试
               │            ┌──────────────────────▼─────────┐
               │            │  8. transform_response::run()  │
               │            │  上游协议 → 出站协议转换       │
               │            └──────────────────────┬─────────┘
               │                                  │
               │            ┌──────────────────────▼─────────┐
               │            │  9. persist_log::run()          │
               │            │  异步发送日志到 channel         │
               │            └──────────────────────┬─────────┘
               │                                  │
               └──────────────────────────────────┤
                                                  │
                                     ┌────────────▼────────────┐
                                     │  finalize::success()    │
                                     │  或 finalize::failure() │
                                     │  构建最终 HTTP Response │
                                     └─────────────────────────┘
```

### 7.2 中间件详细说明

| # | 中间件 | 文件 | 职责 | 副作用 |
|---|--------|------|------|--------|
| 1 | `extract` | `extract.rs` | 解析请求头、URI、方法；读取请求体（上限 2MB）；生成 `request_id` | 写入 `RequestContext` |
| 2 | `authenticate` | `authenticate.rs` | 检查 `/v1/*` 路径的 `Authorization: Bearer <sk-gw-xxx>`；SHA-256 哈希匹配 `gateway_keys` 表 | 注入 `auth_key_name` |
| 3 | `rate_limit` | `rate_limit.rs` | 按客户端 IP + Provider ID 进行请求数/Token 数限流 | 拒绝时返回 429 |
| 4 | `resolve_route` | `resolve_route.rs` | 按 Host/Path/Method/ContentType 匹配 `routing_rules`；解析目标 Provider；支持 `/v1/models` 返回 Provider 模型列表 | 设置 `route`, `provider`, `channels_available` |
| 5 | `select_channel` | `select_channel.rs` | 按 Provider 的 `key_strategy` 从可用 Key 中选择；跳过 `failed_keys` | 设置 `selected_api_key` |
| 6 | `transform_request` | `transform_request.rs` | 调用适配器的 `transform_request()` 实现协议转换；构建上游 URL 和请求体 | 生成 `UpstreamRequest` |
| 7 | `dispatch_upstream` | `dispatch_upstream.rs` | 用 Reqwest 转发请求到上游；支持流式/非流式；指数退避重试（最多 3 次）；SSE 心跳 15s/超时 30s | 调用 `stream_response` |
| 8 | `transform_response` | `transform_response.rs` | 调用适配器的 `transform_response()` 将上游响应转为客户端协议格式 | 更新 `upstream_body` |
| 9 | `persist_log` | `persist_log.rs` | 通过 mpsc channel 异步发送日志 | 写入 `RequestLog` |
| 10 | `finalize` | `finalize.rs` | 构建最终 HTTP Response；设置状态码/响应头/响应体 | 返回 `Response` |

### 7.3 三级失败回退机制

```
Level 1: dispatch_upstream 内部重试（最多 3 次，指数退避 500ms→8s）
    ↓ 失败
Level 2: select_channel 换 Key（跳过已失败的 Key）
    ↓ 所有 Key 失败
Level 3: resolve_route 换渠道（从 channels_available 选下一个 Provider）
    ↓ 所有渠道 + 所有 Key 失败
最终: 返回错误响应 "所有渠道和 Key 均已失败"
```

## 8. 协议适配器系统

### 8.1 ProviderAdapter Trait

```rust
#[async_trait]
pub trait ProviderAdapter: Send + Sync {
    fn provider_type(&self) -> &'static str;

    /// 将原始请求体（JSON）转为上游请求
    async fn transform_request(
        &self,
        req_body: &[u8],
        provider: &Provider,
        selected_api_key: &str,
    ) -> Result<UpstreamRequest, ProtocolError>;

    /// 将上游响应转为客户端期望的格式（JSON）
    async fn transform_response(
        &self,
        resp: &UpstreamResponse,
    ) -> Result<serde_json::Value, ProtocolError>;
}
```

### 8.2 已实现的适配器

| 适配器 | `provider_type` 标识 | `transform_request` 行为 | `transform_response` 行为 |
|--------|----------------------|-------------------------|--------------------------|
| **OpenAI Chat Adapter** | `"openai_chat"` | 原始 OpenAI Chat 格式 → 上游（不变） | 上游响应 → OpenAI Chat 格式 |
| **Claude Messages Adapter** | `"claude_messages"` | 使用 `linguafranca` 转为 Claude Messages 格式 | Claude 响应 → 客户端期望格式 |
| **OpenAI Response Adapter** | `"openai_response"` | 原始 OpenAI Response 格式 → 上游（不变） | 上游响应 → OpenAI Response 格式 |

### 8.3 AdapterRegistry

```rust
pub struct AdapterRegistry {
    adapters: HashMap<String, Box<dyn ProviderAdapter>>,
}

impl AdapterRegistry {
    pub fn new() -> Self;
    pub fn register(&mut self, adapter: Box<dyn ProviderAdapter>);
    pub fn get(&self, provider_type: &str) -> Option<&dyn ProviderAdapter>;
}
```

### 8.4 协议转换流程

```
入站请求 → 根据 route.inbound_protocol 选择 Inbound Adapter
  → UpstreamRequest（URL + Headers + Body）
  → 发送到上游 Provider
  → UpstreamResponse（Status + Headers + Body）
  → 根据 route.outbound_protocol 选择 Outbound Adapter
  → 转换后的响应返回客户端
```

## 12. 认证与限流

### 12.1 网关认证

- **范围**: 仅 `/v1/*` 路径需要认证
- **方式**: `Authorization: Bearer <sk-gw-xxxxx>`
- **验证**: SHA-256 哈希匹配 `gateway_keys` 表
- **Key 生命周期**: 支持过期时间、最大并发、启用/禁用

### 12.2 限流控制

- **粒度**: 按客户端 IP（取 `X-Forwarded-For` 首个 IP）
- **指标**: 每分钟请求数限制 + 每分钟 Token 数限制
- **隔离**: per-provider 独立限流（key = `ip:provider_id`）
- **窗口**: 滑动分钟窗口（60s 重置）
- **存储**: DashMap（全局共享，自动清理过期计数器）

## 13. 完整请求流程

### 13.1 外部请求处理

```
客户端 (curl/Postman/AI Tool)
    │ POST http://127.0.0.1:2013/v1/chat/completions
    │ Authorization: Bearer sk-gw-xxxxx
    │ Content-Type: application/json
    │ {"model":"gpt-4","messages":[{"role":"user","content":"Hello"}]}
    ▼
Axum Router (build_router)
    ├── /health → 200 OK {"status":"ok"}
    └── /* → proxy_handler → GatewayPipeline::execute()
        │
        ├── extract::initialize() + read_body()
        │   → RequestContext { request_id: "uuid", method, uri, headers, body }
        │
        ├── authenticate::run()
        │   → hash_api_key(token) → GatewayKeyRepo::find_by_hash()
        │   → ctx.auth_key_name = Some("key_name")
        │
        ├── rate_limit::run()
        │   → RateLimitState::check(client_ip, tokens)
        │
        ├── resolve_route::run()
        │   → RouteManager::resolve(path, method, content_type)
        │   → 匹配 RoutingRule → Provider → 设置 ctx.route, ctx.provider
        │   → 如果是 /v1/models → 直接返回模型列表响应
        │
        ├── run_with_failover() 循环:
        │   ├── select_channel::run()
        │   │   → 从 Provider.keys 中按策略选一个可用 Key
        │   │
        │   ├── transform_request::run()
        │   │   → AdapterRegistry.get(inbound_protocol).transform_request()
        │   │   → 构建 UpstreamRequest { url, method, headers, body }
        │   │
        │   ├── dispatch_upstream::run()
        │   │   → reqwest::Client::request(method, url)
        │   │   → 设置 Adapter 头 + 转发部分客户端头
        │   │   → 发送请求（流式/非流式）
        │   │   → 重试（最多 3 次，指数退避）
        │   │
        │   ├── 失败时:
        │   │   → 记录 failed_keys
        │   │   → 有可用 Key → 继续 (Level 2)
        │   │   → 无可用 Key → 记录 failed_providers → 换渠道 (Level 3)
        │   │
        │   └── 成功时:
        │       → transform_response::run()
        │       → AdapterRegistry.get(outbound_protocol).transform_response()
        │       → persist_log::run() (异步发送日志)
        │
        └── finalize::success(ctx)
            → 构建 axum Response (status, headers, body)
            → 返回 HTTP Response
```

### 14.3 关键注意事项

1. **布尔字段序列化**: Rust 端使用 `i64`（0/1）存储布尔值，前端通过 `enabled != 0` 或 `enabled: bool` 处理转换
2. **JSON 数组字段**: `Provider.protocols`, `Provider.models`, `Provider.keys`, `ModelMapping.capabilities` 等字段在 Rust 端以 JSON 字符串存储，前端需要 `JSON.parse`/`JSON.stringify`
3. **日期时间**: 传递到前端时转为 `String`（`.to_string()`），前端使用 `new Date()` 解析
4. **加密 Token**: `Provider.keys` 中的 `value` 字段通过 AES-GCM + PBKDF2 加密存储，前端永远不会拿到明文（除了首次创建时）
5. **错误处理**: 所有 Tauri 命令返回 `Result<T, String>`，前端通过 `try/catch` 捕获 `e.message`
