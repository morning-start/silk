# Silk UI 设计总结

## 1. 文档目标

这份文档不是只给出审美方向，而是给后续 `open-design` 或继续 UI 改版时使用的**页面设计基线**。  
目标是明确三件事：

1. Silk 这类产品应该长成什么样
2. 每个页面应该放什么内容，优先级如何
3. 布局、组件、状态、交互应该如何统一

当前项目已经经过多轮 UI 调整，因此后续设计不能只停留在“做一个冷静的控制台”这种抽象描述，而要能直接指导页面重画和组件升级。

## 2. 产品定位

Silk 是一个**纯本地桌面 AI 多模型中转网关**。  
它不是：

- 官网
- 云控制台
- 团队协作后台
- 面向非技术用户的轻娱乐产品

它是一个**本地网关控制台**，核心任务是帮助用户在一台机器上完成：

- 配置 Provider 渠道
- 组织模型映射
- 设置高级路由规则
- 查看请求日志与排障信息
- 控制网关启停与本地行为

因此设计要优先服务：

- 高频操作
- 快速扫描
- 明确状态
- 低学习成本
- 本地可信感

一句话定义：

> Silk 应该像一个 AI Gateway 的本地控制室，而不是一个 AI 产品宣传页。

## 3. 当前产品结构

### 主导航

当前导航结构是合理的，建议保留：

1. 仪表盘
2. 渠道
3. 模型
4. 路由
5. 日志
6. 设置

### 页面职责

| 页面 | 主要职责 | 页面类型 |
|------|----------|----------|
| 仪表盘 | 运行状态总览、快捷启动、最近活动 | 总览页 |
| 渠道 | Provider 资产管理、连通性管理 | 资源管理页 |
| 模型 | 模型映射、渠道绑定、调度信息 | 核心配置页 |
| 路由 | 高级匹配与协议转换 | 高级规则页 |
| 日志 | 请求审计、错误排查、统计观察 | 排障页 |
| 设置 | 网关参数、桌面行为、数据管理、Key 管理 | 系统页 |

这套信息架构已经符合当前产品心智，不建议再人为制造二级结构或营销式首页。

## 4. 现有 UI 基调总结

当前前端已经形成以下基础语言：

- 布局：深色侧栏 + 浅色主工作区 + 顶栏状态区
- 交互：卡片、表格、弹窗、表单为核心
- 信息层级：先摘要，再列表，再详情或弹窗
- 文案：中文、工具化、说明性强
- 数据表现：状态点、Tag、mono 字体、统计卡

### 应该保留的现有优点

- 侧栏与工作区的对比清晰
- 顶栏有全局运行状态和启停操作
- 仪表盘具备“下一步引导”意识
- 渠道与模型页已形成卡片化方向
- 日志页已经偏向高密度工作台而不是空泛报表
- 设置页明确强调本地桌面行为与数据安全

### 当前还不够的地方

- 文档层面缺少页面内容级设计定义
- 视觉方向有基础，但各页面模块关系还不够“系统化”
- 多个页面只是“有卡片”，但还没有明确的卡片类型分级
- 某些页面的布局更像功能拼装，而不是明确的页面编排
- 主色偏 `indigo/purple`，仍带一些通用模板感

## 5. 总体设计方向

### Design Direction

- **Purpose**: 本地 AI Gateway 控制台
- **Audience**: 熟悉模型、API、路由、请求日志的技术用户
- **Tone**: 克制、可信、专业、冷静，略带网络控制台气质
- **Memorable Detail**: “网关在线状态”作为全局视觉锚点，贯穿侧栏、顶栏、仪表盘和关键卡片
- **Constraint**: 桌面应用、高信息密度、中文文本、延续 Naive UI 组件生态

### 关键词

- Local-first
- Gateway control room
- Operational clarity
- Structured density
- Trusted tooling

### 不要走的方向

- 不要做成官网首屏风格
- 不要做成玻璃拟态展示板
- 不要把每页都做成大英雄卡 + 几个装饰数字
- 不要让“品牌表达”压过“操作效率”

## 6. 全局布局方略

## 6.1 布局骨架

全局结构建议固定为三层：

1. **左侧导航层**
2. **顶部状态层**
3. **主工作区**

### 左侧导航层

用途：

- 页面切换
- 品牌识别
- 当前运行实例的静态信息

应保留：

- 产品名
- 版本 / 产品副标题
- 分组导航
- 当前监听地址或实例标识

不应堆太多动态操作。  
动态控制应主要放在顶栏和页面主体。

### 顶部状态层

用途：

- 当前页面标题
- 网关运行状态
- 启停 / 重启
- 主题切换

这是全局操作区，不应被挤成纯装饰头部。  
需要始终保持“用户看一眼就知道网关是否活着”。

### 主工作区

每个页面建议统一使用以下结构：

1. 页面头部
2. 页面摘要区
3. 主内容区
4. 次要内容区 / 详情区 / 分页区

其中页面头部至少包含：

- 页面标题
- 数量标签或状态摘要
- 主操作按钮

## 6.2 页面网格规则

推荐统一使用桌面优先布局：

- 页面横向内容宽度：铺满可用区域，不做营销式窄容器
- 常规间距：`16 / 20 / 24`
- 卡片圆角：`12~16`
- 统计卡：一行 `4` 或 `5`
- 主次布局：`2:1`、`3:1`、`2:3` 三种比例优先

建议形成统一页面模板：

### 模板 A：总览页

- 顶部摘要卡
- 中部统计卡行
- 下部主内容 + 侧边操作

适用：仪表盘

### 模板 B：资源卡片页

- 顶部工具栏
- 中部搜索 / 过滤
- 下部卡片网格

适用：渠道、模型

### 模板 C：高密度表格页

- 顶部指标摘要
- 中部筛选工具栏
- 下部表格
- 底部分页 / 详情弹窗

适用：日志、部分路由

### 模板 D：系统配置页

- 顶部说明
- 多个分组卡片
- 每卡一类设置或数据操作

适用：设置

## 6.3 页面头部规范

每页头部统一结构：

- 左侧：标题 + 数量 / 状态标签
- 右侧：主操作按钮 + 次操作

不要每页都重新发明头部。  
这样可以让所有页面看起来属于同一产品，而不是多个独立 demo 页。

推荐层级：

- 一级标题：18~22px
- 数量 Tag：小号、低干扰
- 主操作按钮：1 个
- 次级按钮：0~3 个

## 7. 全局视觉系统建议

## 7.1 色彩

推荐从当前 `indigo` 主色转向更符合“网关 / 网络 / 控制”语义的体系：

- 基础中性色：`Slate`
- 主强调色：`Cyan` 或偏冷蓝
- 成功色：`Emerald`
- 警告色：`Amber`
- 错误色：`Rose/Red`

### 状态色语言要固定

| 状态 | 颜色方向 | 含义 |
|------|----------|------|
| 运行中 | Emerald | 正常在线 |
| 检测中 | Amber | 等待 / 刷新 / 未稳定 |
| 异常 | Red | 失败 / 错误 |
| 禁用 | Slate Gray | 非激活 |
| 连接 / 网络信息 | Cyan | 地址 / 端口 / 链接 |

状态色一旦定义，不要在不同页面随意改语义。

## 7.2 字体与数字

普通文本可以继续使用系统无衬线，但以下信息应统一 mono：

- 地址
- 端口
- 路径
- 模型 ID
- request id
- token / cost / duration / count

数字建议统一 `tabular-nums`，这样日志和统计页的扫描效率会明显更高。

## 7.3 卡片分级

当前产品需要明确三类卡片：

### 1. 概览卡

用于：

- 今日请求数
- 平均响应
- token 消耗
- 活跃渠道

特征：

- 数值大
- 说明短
- 一眼可扫

### 2. 资源卡

用于：

- Provider
- 模型映射

特征：

- 标题明确
- 中间是关键信息块
- 底部是轻操作区

### 3. 系统卡

用于：

- 设置项
- 数据导入导出
- Key 管理

特征：

- 结构稳定
- 说明性强
- 操作按钮明确

## 8. 交互原则

### 8.1 状态优先

用户第一眼先确认：

- 网关是否在线
- 页面数据是否为空
- 是否存在错误
- 当前页面是否可操作

因此错误态、空态、运行态不能只是组件默认样式，应该有统一视觉表达。

### 8.2 快捷操作优先

Silk 是操作型工具，不是阅读型产品。  
页面优先级应向这些动作倾斜：

- 启动 / 停止 / 重启
- 测试 Provider
- 复制地址
- 新增配置
- 导出日志
- 查看详情

### 8.3 复杂编辑要分段

弹窗模式可以保留，但复杂表单必须有阶段感。  
尤其是模型映射这种核心配置：

1. 选择渠道
2. 选择模型
3. 设置策略和能力
4. 补充价格 / token / 描述

不要让复杂配置在视觉上变成一整块平铺输入框。

### 8.4 空状态必须给动作

空状态需要同时回答：

- 现在为什么是空的
- 接下来应该做什么
- 用户能否立刻开始

### 8.5 高级功能要“可见但不吓人”

路由页就是典型高级页。  
应该让用户知道它存在，但不要让普通用户觉得非配不可。

## 9. 页面内容与布局蓝图

以下部分是最重要的，直接定义每页应该包含哪些模块，以及推荐的布局关系。

## 9.1 仪表盘

### 页面目标

- 告诉用户 Silk 当前是否可用
- 给出最快启动路径
- 展示当天最核心的使用情况
- 提供最近请求与网关控制入口

### 首屏内容优先级

1. 运行状态 / 欢迎区
2. 核心指标卡
3. 最近请求
4. 网关控制
5. 首次使用引导

### 推荐布局

第一行：全宽欢迎 / 状态卡  
第二行：4 个统计卡  
第三行：左 `2/3` 最近请求，右 `1/3` 网关控制  
可选第四行：首次使用 checklist / 最近异常摘要

### 欢迎区内容建议

运行中时显示：

- 当前监听地址
- 今日请求数
- 快捷按钮：复制地址 / 查看日志 / 管理渠道

未运行时显示：

- 当前未启动说明
- 推荐下一步
- 主按钮：启动网关
- 次按钮：去配置渠道

### 统计卡建议固定为

- 今日请求数
- 平均响应时间
- 今日 Token 消耗
- 活跃渠道数

不要在仪表盘塞过多 BI 信息。  
它是状态总览，不是分析中心。

### 改版重点

- 欢迎卡要减少“营销感渐变”，更像运行状态面板
- 最近请求表格要突出状态码、模型、耗时
- 网关控制卡可以更像“设备控制面板”

## 9.2 渠道页

### 页面目标

- 快速查看所有 Provider 状态
- 快速新增、测试、编辑、删除
- 快速判断某个 Provider 是否能用于生产请求

### 当前方向判断

当前使用卡片网格是正确方向，不建议改回表格主视图。  
因为用户更关心“对象状态”，而不是一堆紧凑字段。

### 页面内容结构

1. 页面头部：标题、数量、搜索、新增按钮
2. 过滤区：搜索，可预留状态筛选 / 协议筛选
3. Provider 卡片网格
4. 空状态 / 错误状态
5. 新增 / 编辑弹窗

### 每张 Provider 卡片应包含

- 名称
- 在线状态 / 健康状态
- API 域名或 base url 摘要
- key 数量
- 模型数量
- 超时 / 重试等轻元信息
- 操作：编辑 / 测试 / 删除

### 推荐布局方略

- 桌面下 `3` 列卡片
- 宽屏可扩展到 `4` 列
- 卡片高度尽量统一
- 操作区固定在底部

### 需要强化的内容

- “健康状态”和“启用状态”要区分视觉语义
- URL 区域要像技术信息，不要像普通正文
- 测试结果可以考虑让状态短暂写回卡片，而不是只弹 toast

### 弹窗内容结构建议

分为四段：

1. 基本信息
2. 协议与地址
3. Key 策略与 Key 列表
4. 模型同步与连接参数

## 9.3 模型页

### 页面目标

- 让用户把外部请求模型名与内部真实渠道能力对齐
- 展示模型映射的调度关系
- 让模型配置从“表单”升级为“调度面板”

### 页面定位

这是 Silk 的核心特色页之一。  
它不应该只是“另一个 CRUD 页面”，而应突出 Silk 的中转和路由价值。

### 页面内容结构

1. 页面头部：标题、数量、新增按钮
2. 模型映射卡片网格
3. 空状态 / 错误状态
4. 新增 / 编辑弹窗

### 每张模型卡建议包含

- 模型映射名
- 启用状态
- 描述
- 绑定渠道数量
- 渠道摘要 badge
- token 上限信息
- 价格信息
- 能力标签
- 操作：编辑 / 删除

### 推荐布局方略

- 卡片网格为主
- 每卡需要明显分区：
  - 标题区
  - 渠道区
  - 规格区
  - 价格区
  - 能力区
  - 操作区

### 编辑弹窗建议升级为“分段式”

建议至少按以下顺序呈现：

1. 模型名
2. 关联渠道
3. 从渠道中选择模型
4. 负载策略
5. token 与价格
6. 描述与能力
7. 启用状态

### 当前页面的核心设计任务

- 把“调度关系”做得比“输入框列表”更清晰
- 让用户能一眼看出一个模型会落到哪些渠道
- 让能力、价格、上下文信息形成结构化辅助，而不是噪音

## 9.4 路由页

### 页面目标

- 让高级用户精确控制匹配路径、方法、协议转换和目标渠道
- 给普通用户清楚提示：这不是默认必配项

### 页面内容结构

1. 页面头部：标题、数量、新增按钮
2. 顶部说明 Alert
3. 快速模板卡
4. 规则列表
5. 新增 / 编辑弹窗

### 推荐布局

当前结构基本正确，建议继续沿用：

- 说明区在上
- 模板区其次
- 主数据区放表格

### 为什么路由页用表格更合适

路由规则是“条件 -> 目标”的结构化规则，不是资源对象。  
因此比卡片更适合表格：

- 便于横向比较
- 便于看路径、方法、目标、转换状态
- 更像真正的规则管理器

### 表格列建议保持

- 名称
- 匹配路径
- 方法
- 目标渠道
- 协议转换
- 启用状态
- 操作

### 需要加强的点

- 表格上方可增加“优先级说明”或排序语义提示
- 弹窗里可加入“规则将如何生效”的帮助文字
- 模板卡可以更像“快速创建入口”，而不是普通按钮堆

## 9.5 日志页

### 页面目标

- 成为排障中心
- 允许用户快速定位失败请求、性能问题、成本问题
- 支撑“查看详情 -> 判断原因 -> 回到配置”的工作流

### 页面定位

日志页是高密度工作页，不应过度追求视觉留白。  
这里比美观更重要的是扫描效率。

### 页面内容结构

1. 顶部指标卡
2. 页面头部与筛选工具栏
3. 主日志表格
4. 渠道占比摘要
5. 分页区
6. 详情弹窗

### 推荐布局

第一行：5 个指标卡  
第二行：标题 + 筛选 / 操作  
第三行：日志主表格  
第四行：渠道摘要  
底部：分页  
弹窗：日志详情

### 表格是核心，不要弱化

表格应该是页面视觉主角。  
重点列：

- 时间
- 方法
- 路径
- 状态
- 响应
- 耗时
- Token
- 费用
- 模型
- 渠道
- 重试
- 流式

### 详情弹窗应该回答的问题

- 这次请求是谁发的
- 走了哪个路径
- 命中了哪个模型和渠道
- 是否经过协议转换
- 响应状态如何
- 耗时、token、cost 怎么样

### 页面改版重点

- 指标卡与日志表格之间的层级要更明显
- 筛选工具栏要更像“操作台”
- 错误状态行可以考虑更强的视觉强调
- 详情弹窗可以采用双栏信息布局，提高阅读效率

## 9.6 设置页

### 页面目标

- 管理网关基础设置
- 管理本地桌面行为
- 管理配置 / 数据导入导出
- 管理 Gateway Key

### 页面结构

1. 页面头部：标题 + 保存按钮
2. 顶部说明 Alert
3. 网关基础卡
4. 桌面行为卡
5. 配置与数据卡
6. Key 管理卡

### 页面类型判断

设置页不是单张超长表单，而是“多区块系统页”。  
当前思路是正确的，应继续强化为：

- 一张卡只处理一类任务
- 每张卡都有明确标题和说明
- 危险操作显式标注

### 需要强调的体验

- 本地性
- 可恢复
- 安全感
- 操作有后果说明

### Key 管理的设计要点

- 新建 Key 后的一次性展示必须足够醒目
- 复制动作要非常直接
- 已有 Key 列表要一眼看出启用状态
- 危险操作（删除）要明显但不抢主流程

## 10. 组件级设计要求

## 10.1 Toolbar

统一包含：

- 左：标题 / 标签
- 右：搜索 / 筛选 / 操作

所有页面优先复用统一 toolbar 结构，不要每页自定义风格。

## 10.2 Empty State

空状态统一需要：

- 图标
- 标题
- 原因说明
- 直接动作按钮

## 10.3 Error State

错误状态统一需要：

- 错误图标
- 失败标题
- 错误说明
- 重试按钮

## 10.4 Modal

弹窗统一要求：

- 标题清晰
- 表单分组明显
- footer 操作一致：取消 / 确认
- 复杂弹窗要有段落标题或视觉区隔

## 10.5 Tag / Status

Tag 不只是装饰，应该承担以下语义：

- 启用 / 禁用
- 健康 / 异常
- 方法类型
- 能力标签
- 运行状态

不同页面内同语义的 Tag 应保持同色。

## 11. 响应式与桌面适配策略

Silk 主要是桌面应用，但仍需考虑窄宽度和窗口缩放。

### 桌面优先原则

- 默认针对宽屏桌面设计
- 不需要像移动端产品一样大改信息架构
- 但不能在窗口缩小时立刻崩掉

### 最低要求

- 卡片网格可从 4 -> 3 -> 2 列缩减
- 顶栏按钮不能因缩窄而错位
- 表格页必须支持横向内容压缩或滚动
- 弹窗在较小窗口下仍能完整滚动

## 12. 对 `open-design` 的具体输出要求

如果把后续设计任务交给 `open-design`，不应只要求“出一个设计方向”，而应要求以下交付物：

1. 整体设计主题与视觉关键词
2. 全局色板、状态色、背景层级定义
3. 侧栏 / 顶栏 / 内容区布局规范
4. 页面头部与 toolbar 统一规范
5. 仪表盘高保真布局稿
6. 渠道页卡片规范
7. 模型页“调度面板式”卡片和弹窗方案
8. 路由页表格化规则管理方案
9. 日志页高密度表格与详情弹窗方案
10. 设置页系统卡片规范
11. 空状态 / 错误状态 / 加载状态统一规范
12. 窄宽度窗口下的退化规则

重点是：  
它产出的必须是**可以直接指导页面改造的页面蓝图**，而不是只给 moodboard 或抽象风格词。

## 13. 最终结论

Silk 的 UI 不应该继续往“泛 AI 产品模板”靠，而应该明确收敛成一套**本地网关控制台设计系统**。  
设计重点不只是“更好看”，而是：

- 让状态更清楚
- 让结构更统一
- 让配置更高效
- 让排障更直接
- 让模型调度能力更可见

后续所有页面改版，都建议围绕四个核心词判断是否正确：

**可信、清晰、高频操作、控制感。**
